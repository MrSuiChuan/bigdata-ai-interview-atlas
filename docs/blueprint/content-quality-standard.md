# 一句话结论

本系统以后不再用数量判断内容完成度。一个主题只有同时满足结构、来源、原理深度、题库转化、样例质量和前端可见性，才能算真正达标。

# 通用标准

1. 必须用中文解释，专业术语可以保留英文。
2. 必须说明核心对象、状态、交互、生命周期。
3. 必须说明完整链路，而不是只列名词。
4. 必须说明系统保证什么、不保证什么。
5. 必须说明常见故障、排查入口和恢复边界。
6. 必须说明工程权衡，不能只给结论。
7. 必须关联来源、Claim 和题库。
8. 必须能在前端被浏览、搜索和练习。

# 大数据组件达标标准

每个大数据组件至少要覆盖：

1. 总览定位。
2. 核心对象与状态。
3. 架构与角色分工。
4. 写路径。
5. 读路径。
6. 元数据与状态管理。
7. 故障与恢复。
8. 一致性和不保证项。
9. 性能模型。
10. 分区、切分、布局或数据组织。
11. 后台维护任务。
12. 资源治理。
13. 安全与权限。
14. 可观测性。
15. 故障排查。
16. 配置与调优。
17. 系统设计场景。
18. 横向对比。
19. 面试答题手册。
20. 真实或准真实样例。

# AI / LLM 达标标准

AI 相关主题至少要覆盖：

1. 概念边界。
2. 核心机制。
3. 工程链路。
4. 框架或协议边界。
5. 生产风险。
6. 观测和评估。
7. 安全问题。
8. 项目复盘表达。
9. 实践资料 实践来源 来源映射。
10. 官方来源交叉复核状态。

# 实践资料 专区达标标准

实践资料 项目不能直接搬运，必须完成：

1. 来源登记。
2. 项目定位。
3. 适用人群。
4. 核心知识点提炼。
5. 可转化面试问题。
6. 和主知识库映射。
7. 需要官方复核的点。
8. 导入状态标记。

# Claim 标准

1. Claim 必须是事实判断，不是写作建议。
2. Claim 必须尽量原子化。
3. Claim 必须有来源。
4. generated claim 必须人工复核后才能 high confidence。
5. 仅社区来源 Claim 必须标明 实践来源 适用范围。

# 样例标准

1. 概念性样例必须明确标注 conceptual。
2. 可运行样例必须说明运行命令和依赖。
3. 需要集群的样例必须说明前提条件。
4. SQL 样例必须说明适用引擎。
5. 样例不能只复述概念，要体现真实操作或排查链路。

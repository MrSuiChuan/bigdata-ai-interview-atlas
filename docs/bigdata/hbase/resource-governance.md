## HBase 的资源治理，首先不是配额问题，而是模型问题
很多系统的资源治理可以主要靠队列、CPU 配额或资源池来做，但 HBase 不一样。它的资源消耗很大程度上是由表模型直接决定的：

- `RowKey` 决定热点。
- 列族决定物理读写分组。
- 版本与 TTL 决定长期状态膨胀。
- Scan 模式决定 IO 和缓存压力。
- Region 数量决定元数据和调度开销。

所以 HBase 的资源治理如果只从“限制谁用多少”入手，而不先控制表设计和访问模式，往往治不到根上。

## 资源治理真正要守的边界
更成熟的 HBase 治理，不是先问“怎么分机器”，而是先问“哪些行为不该让它进来”。下面这张表最能说明治理边界在哪里：

| 治理边界 | 不治理会怎样 | 更有效的动作 |
| --- | --- | --- |
| 键模型边界 | 热点长期压在少数 Region/RegionServer 上 | 上线前审查 `RowKey`、热点前缀、打散策略 |
| 访问边界 | 大 scan 抢占缓存、磁盘和 compaction 资源 | 约束在线表查询模式，把分析负载导向分析系统 |
| 存储边界 | 列族、版本、TTL 无序膨胀 | 设列族上限、版本基线、TTL 基线 |
| 区域边界 | Region 过多或过偏，恢复和调度成本持续升高 | 控制 Region 数量与增长节奏，重点治理热表 |
| 恢复边界 | 热表出事时只能依赖线上表自救 | 单独规划热点表的恢复、备份和复制策略 |

## 多租户下最常见的资源失控源头
### 热点表拖垮局部节点
一个租户如果键分布不均，可能把流量长期打到少数 RegionServer，形成局部资源挤占。

### 大 scan 持续抢占 IO
一些分析式访问如果直接落在 HBase 上，可能用长时间 scan 占满磁盘与缓存资源，影响在线点查。

### 版本与删除标记堆积
某些表长期保留过多版本或产生大量删除标记，会持续提高 compaction 和读放大成本。

### 过多列族或 Region
列族设计过细、Region 切分过密，都会带来额外常驻管理负担。

### 把在线表当分析表
这是最常见也最“看起来合理”的误用。业务觉得数据就在 HBase 里，于是直接做大范围扫描、批量导出、离线校验，结果在线点查没有明显增加，集群却开始出现缓存被穿透、compaction 债务上升和长尾延迟。

## 真正的资源边界应该怎么划
在 HBase 里，更有效的治理边界往往是：

1. 表级边界：哪些业务能建表、怎么建表、列族上限和版本策略是什么。
2. 访问边界：哪些场景允许 scan，哪些必须经由上层索引或导出分析系统。
3. 热点边界：热点键模型是否允许上线，是否有预防性审查。
4. 长期状态边界：TTL、版本、删除策略、快照和备份如何控制。

也就是说，资源治理不能只看运行期，还必须前移到建模与评审阶段。

## 为什么“单表治理”比“集群平均治理”更重要
HBase 很容易出现“平均指标看起来没问题，但单表已经把局部节点压垮”的情况。因此治理时应该优先关注：

- 哪些表最热。
- 哪些表 Region 最偏。
- 哪些表 compaction 债务最高。
- 哪些表 scan 最重。

这也是为什么 `hbtop`、热点 Region 视角和表级审计，比只看全局 CPU 平均值更有治理意义。

## 一个典型的失控链
很多线上事故并不是瞬间出现，而是按下面的顺序慢慢积累出来：

1. 业务建表时没有约束键模型和列族策略。
2. 某张表开始承接越来越多的混合访问。
3. 热点、版本、删除标记和 scan 慢慢堆积。
4. flush/compaction 背景成本升高，主请求延迟开始抖。
5. 团队只看到“机器更忙了”，却没看到其实是单表治理早已失守。

资源治理的价值，就是尽量在第 1 到第 3 步就把问题挡住，而不是等第 5 步再扩机器。

## 一些典型治理动作
- 控制列族数量，避免无意义拆分。
- 设定版本与 TTL 基线，不允许无限增长。
- 对热点键模型做上线前审查。
- 限制把 HBase 当分析引擎直接跑大 scan 的行为。
- 对超热表做单独容量与恢复设计。

## 上线前至少要问的四个问题
1. 这张表的主请求到底是不是围绕 `RowKey` 或前缀展开。
2. 有没有明显会形成租户热点、时间热点或批任务热点的模式。
3. 版本、TTL、列族和 Region 规模有没有明确上限。
4. 如果这张表成为热表，恢复目标和下游分析边界是否已经提前设计。

## 本页结论
HBase 的资源治理核心不在“怎么平均分机器”，而在“怎么防止某张表、某种访问模式或某类键模型破坏全局稳定性”。真正有效的治理，必须把表设计、访问边界和运行期观测放在一起看。

## HBase 安全不是一个开关，而是一套分层边界
很多人会把 HBase 安全简化成“有没有 ACL”或者“有没有 Kerberos”。这种说法不够完整。更准确的理解方式是：HBase 安全是多层共同组成的，至少包括：

- 身份认证：谁在访问。
- 传输保护：请求在网络上是否被保护。
- 授权控制：这个身份能访问哪些表或列族。
- 底层存储边界：HDFS 层是否也受控。
- 运维审计：发生过什么操作，能不能追溯。

## 先把安全层次画对
如果把安全边界画成一张矩阵，理解会稳定很多：

| 层次 | 要回答的问题 | 如果缺失会怎样 |
| --- | --- | --- |
| 身份认证 | 访问者到底是谁 | 权限和审计都没有可信主体 |
| 传输保护 | 请求在网络上传输时是否受保护 | 身份可信但链路不可信，仍可能泄露或被劫持 |
| 授权控制 | 这个身份能读什么、写什么、管理什么 | 认证通过后仍可能越权操作 |
| 底层存储权限 | 即使绕开表入口，底层文件是否仍受控 | “表面安全，底层裸奔”的假隔离最常见 |
| 运维审计 | 谁在何时对哪张表做了什么 | 事后无法复盘，治理无法闭环 |

## 身份认证边界：先回答“你是谁”
在生产环境里，HBase 通常不是匿名服务。Kerberos 等认证机制的意义，是先把客户端、服务进程和运维动作绑定到明确身份上。

如果身份都不可信，后面的 ACL 与审计就没有坚实基础。因此安全问题的第一层永远是认证，而不是先问某张表权限怎么配。

## 授权边界：不是“连上就能看所有表”
认证通过只回答了“你是谁”，没回答“你能干什么”。HBase 的 ACL 和授权边界通常围绕：

- 命名空间。
- 表级。
- 某些更细粒度访问边界。

这里最重要的是边界意识：HBase 安全治理不能建立在“默认大家都可信”的团队习惯上，而要建立在明确权限分层之上。

## 底层 HDFS 权限仍然是安全链的一部分
HBase 虽然提供表级和服务级访问边界，但它的数据最终仍落在 HDFS 等底层存储上。也就是说：

- HBase 层权限不能脱离底层存储权限理解。
- 底层服务账号、目录可见性和运维权限同样重要。
- 只做应用层控制，不做底层存储控制，边界是不完整的。

## 最常见的假安全，是边界不一致
真正出问题时，往往不是“什么都没做”，而是只做了一半：

1. catalog 不可见，但底层文件仍可直接读取。
2. 应用账号没有写权，但共用高权限服务账号仍能跨表写入。
3. 表级 ACL 配了，但危险运维动作没有审计，事后无法追责。

所以安全治理不能只看某一层“有没有开”，而要看多层边界是否一致。

## 传输与服务端保护为什么也要单独看
即使权限配得对，如果 RPC 通道、服务间传输或关键管理接口缺乏保护，风险依然存在。所以安全不能只停留在“逻辑表权限”，还要看：

- 服务通信是否受保护。
- 管理接口是否暴露过宽。
- 敏感配置和凭据是否被正确管理。

## 审计为什么是治理而不是附加项
安全治理最怕的一类问题是“事后根本说不清谁做了什么”。因此审计并不是锦上添花，而是闭环：

- 谁访问了哪张表。
- 谁做了结构变更。
- 谁触发了危险运维动作。
- 出问题后能否快速还原上下文。

没有审计，很多权限问题即使临时拦住了，也难以长期治理。

## 安全排查时至少要留的证据
1. 身份证据：谁在以什么身份访问集群和表。
2. 授权证据：命名空间、表级和更细粒度边界到底配成了什么。
3. 底层证据：底层存储目录、服务账号和运维账号是否匹配最小权限。
4. 审计证据：关键读写、结构变更、恢复操作是否能追溯到人和作业。

## 安全边界题最容易答错的地方
1. 只说 Kerberos，不说授权和底层存储边界。
2. 只说 ACL，不说身份认证前提。
3. 把 HBase 权限和业务应用权限完全混为一谈。
4. 忽略运维与审计链。

## 本页结论
HBase 安全不是单点功能，而是一条分层边界链：认证、传输保护、授权、底层存储权限和审计缺一不可。只要把这几层说清，你对 HBase 安全治理的理解就已经超出“会配权限”的层面了。

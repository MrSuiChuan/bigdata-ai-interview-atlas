# 题目

什么时候应该继续用 n8n，什么时候应该把开放式子任务交给更重的 Agent Runtime？

# 一句话结论

显式业务流程和系统集成仍由 n8n 主导；一旦任务开始依赖长运行状态机、复杂恢复或多智能体共享状态，就要考虑把开放式部分交给更重 runtime。

# 核心机制

1. n8n 强在事件驱动与系统集成。
2. 更重 runtime 强在长运行状态、恢复和开放式规划。
3. 选型分水岭在复杂度主导权，而不是是否接入 AI。
4. 混合架构通常更稳。

# 标准答案

如果任务主要是由 Trigger 驱动、流程路径相对显式、目标是把 AI 嵌入到一条业务自动化链路里，n8n 就是合适选择。但如果系统开始需要长运行状态机、正式 checkpoint、复杂人机中断、多智能体共享状态或更高强度的运行时治理，那么应把开放式子任务交给更重的 Agent Runtime，让 n8n 继续承担上层集成和业务事件编排。这种“workflow 主链 + runtime 子任务”的混合模式，通常比强行用单一平台承接全部复杂度更稳。

# 必答点

1. 说明 n8n 的长处。
2. 说明更重 runtime 的补位能力。
3. 说明复杂度主导权是分水岭。
4. 说明混合架构比一刀切更合理。
5. 说明这不是平台高低，而是责任分工。

# 常见误答

1. 只说“都能做”。
2. 不讲恢复和共享状态。
3. 认为接入 AI 就必须全盘 Agent 化。
4. 不讲混合架构。

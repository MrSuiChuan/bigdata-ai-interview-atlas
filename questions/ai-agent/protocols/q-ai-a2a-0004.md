# 题目

A2A 和 MCP 在生产架构里如何分层，为什么不能互相替代？

# 一句话结论

MCP 管工具、资源和 prompt 的能力接入，A2A 管远端 Agent 系统之间的协作合同、任务生命周期和信任边界；二者是互补关系。

# 核心机制

1. MCP 是能力暴露协议
2. A2A 是远端 Agent 协作协议
3. Agent Card、Task 生命周期、input-required 和 subscription 是 A2A 的关键语义

# 标准答案

A2A 和 MCP 不能互相替代，因为它们解决的是不同层次的问题。MCP 面向 tools、resources、prompts，解决 Agent 如何接内部或外部能力；A2A 面向 remote agent collaboration，解决一个 Agent 系统如何发现、认证并协作另一个 Agent 系统。A2A 的 Agent Card 推荐发布在 `/.well-known/agent-card.json`，用于说明 identity、service URL、capabilities、authentication schemes 和 skills metadata，是远程协作合同入口。敏感场景下 discovery 不能默认全公开，应使用 authenticated extended cards 或受控 endpoint。执行层面，A2A 是 async-first，围绕 task 生命周期建模，状态包括 submitted、working、completed、failed、canceled、input-required、rejected；`input-required` 是续跑边界，客户端应使用相同 `taskId` 和 `contextId` 继续推进原任务。Subscription 让客户端从当前 Task 开始一致观察非终态任务直到终态。生产架构里可以内部用 MCP 接工具，对外用 A2A 暴露 Agent 协作能力。

# 必答点

1. 说明 MCP 和 A2A 的对象不同
2. 说明 Agent Card 是协作合同入口
3. 说明 discovery 要分层披露
4. 说明 task 生命周期和 `input-required`
5. 说明 subscription 是状态一致观察，不是普通打字机流

# 常见误答

1. 说 A2A 是远程工具调用协议
2. 说 A2A 会替代 MCP
3. 把 Agent Card 当介绍页
4. 不讲 task 状态机
5. 不考虑企业信任边界

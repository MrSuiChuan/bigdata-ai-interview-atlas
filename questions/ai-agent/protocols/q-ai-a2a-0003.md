# 题目

为什么 A2A 的核心不是远程 RPC，而是 Agent Card、Task 生命周期和信任边界？

# 一句话结论

因为 A2A 面向的是“远程协作主体”而不是“远程函数”，它必须先解决发现、协商、续跑和受信任披露，之后才谈得上调接口。

# 核心机制

1. A2A 管远程 agent 协作，MCP 管内部能力接入
2. Agent Card 是远程协作合同而不只是名片
3. task 状态机和 subscription 定义了长任务与人工续跑语义

# 标准答案

A2A 的核心不是远程 RPC，因为它建模的对象不是一个函数接口，而是一个可发现、可持续协作的远程 agent。官方 A2A and MCP 文档已经明确指出，A2A 用于 agent 间协作，MCP 用于工具和资源接入，所以 A2A 的重点是协作主体之间的合同和任务推进，而不是内部能力怎么接。发现层面，A2A discovery 推荐把 Agent Card 发布在 `/.well-known/agent-card.json`，其中不仅有 identity 和 service URL，还有 capabilities、authentication schemes、skills 的输入输出模式和示例，因此 Agent Card 本质上是远程协作合同入口。安全层面，官方还建议对敏感信息使用 authenticated extended cards 或受控 endpoint，说明 discovery 必须分层披露。执行层面，A2A 是 async-first 的，基于 HTTP、JSON-RPC 2.0 和 SSE，围绕 task 建模而不是单次调用建模；`TaskStatus` 有 `state`、可选 message 和时间戳，状态包括 submitted、working、completed、failed、canceled、input-required、rejected。尤其是 `input-required`，它不是普通错误，而是明确的中断态，客户端应使用相同的 `taskId` 和 `contextId` 继续推进原任务。再加上 subscription 只能订阅非终态 task，且首个事件必须是当前 `Task`，终态后流关闭，这使远程长任务具备一致的观察与续跑语义。所以真正成熟的 A2A 设计，必须一起考虑 discovery、task 状态机和信任边界，而不是把它当成一个“带 SSE 的 JSON-RPC”。

# 必答点

1. 说明 A2A 和 MCP 解决的问题层次不同
2. 说明 Agent Card 是远程协作合同的一部分
3. 说明 `input-required` 是续跑边界，不是普通失败
4. 说明 subscription 的价值是状态一致观察，而不只是实时推送

# 常见误答

1. 把 A2A 说成“agent 调 agent 的 RPC”
2. 把 discovery 看成纯展示信息
3. 不区分公开 card 和受控 discovery
4. 忽略 task 生命周期对长任务协作的重要性
